# CLAUDE.md — グラスト G-DX 共通開発ルール（GRUST Common Rules）

このファイルは、**株式会社グラスト（GRUST.inc）が開発するすべての業務システム**をClaude Code等のAIエージェントで開発・改修する際の**最上位ルールブック**です。
G-DX配下の全プロダクト（g-dx_kanri / G-HR / Luxates / RGD / dreaMs / G-PO ほか今後新設するシステム）に横断適用されます。

各システム固有のルール（要件定義の参照先・ロール定義など）は、各リポジトリの `CLAUDE.md` で本ドキュメントを参照したうえで追記してください。
**本ドキュメントおよび以下の絶対参照ドキュメントと矛盾する実装やルール変更は一切禁止します。**

---

## 0. 絶対参照ドキュメント（MUST READ）

今後のシステム制作やプロトタイプ作成にあたっては、必ず以下の2つのドキュメントを**最優先で参照・厳守**してください。

1. **[G-DX 開発・デザインルール（Future Edition）](./G-DX_Future_Design_Rules.md)**
   - 目的: 「生成AI感のないプロフェッショナルな業務UI」の実現
   - 内容: ホワイト/ブラックテーマ、フォントサイズ微調整、サイドバー開閉、UX重視ローディング、Supabase標準化、絶対禁止事項（過剰な角丸・シャドウ・グラデーション・パステルカラーの禁止）など。
2. **[G-DX Lark連携・実装ルール（Future Edition）](./G-DX_Lark_Integration_Rules.md)**
   - 目的: LarkとG-DXシステムのシームレスなUXと運用保守性の担保
   - 内容: Larkプロフィール画像のシステム内表示、連携設定のデータベース管理（ハードコード禁止）、カスタムボット追加の注意書き、非同期通知のベストプラクティスなど。

---

## 1. 最重要原則（必ず守る）

1. **"生成AIっぽさ"を出さない**
   いずれのシステムも業務用SaaSとして長期運用されます。流行のビジュアル（ネオンカラーのグラデーション、過剰な `rounded-2xl`、ガラス風、ネオモーフィズム、紫ピンクの配色、過剰なアニメーション、巨大ヒーロー）は一切使わず、**日本のSaaSデザインを意識した落ち着いたUI**にしてください。
2. **デザイントークンの外の値を一切書かない**
   HEX、px、フォントサイズを直書きしないでください。`G-DX_Future_Design_Rules.md` で定義されたCSS変数を必ず参照してください。
3. **Lark連携設定をハードコードしない**
   Larkの各種ID（Base ID、Table ID、チャットID等）は必ずシステム内のデータベースで管理し、UIから変更可能にしてください。
4. **既存MVP思想（履歴保持・監査ログ必須・運用負荷優先）を壊さない**
5. **絵文字をUIに出さない**

## 2. 技術スタック（標準）

新規システムは以下を**標準スタック**とします。逸脱する場合は事前にユーザー承認が必要です。

| レイヤー | 採用技術 |
|---|---|
| フレームワーク | **Next.js 15（App Router）** 最優先、代替: Vite + React |
| 言語 | **TypeScript**（strict mode、`any` 原則禁止） |
| UI | React 19 |
| スタイリング | **Tailwind CSS v4**（G-DXテーマを適用） |
| フォント | **Noto Sans JP**（Google Fonts）のみ |
| **データベース** | **Supabase（PostgreSQL）** を標準とする。軽量用途はLark Baseで代替可（ユーザー指示に従う） |
| **認証** | **Lark OAuth 2.0** または **ID + パスワード**（2パターン対応） |
| UIコンポーネント | shadcn/ui（G-DXトークンでカスタマイズ） |
| アイコン | **Lucide React** のみ（他アイコンライブラリの混在禁止） |
| データ取得 | Server Components / Server Actions を第一選択、必要時のみ TanStack Query |

## 3. AIエージェントへの進め方の指示

### 3.1 作業着手時にやること

1. 必ず本ファイルと同じディレクトリにある `G-DX_Future_Design_Rules.md` と `G-DX_Lark_Integration_Rules.md` を読み込むこと。
2. 各システムの `CLAUDE.md` と要件ドキュメントを読むこと。
3. 影響するテーブル・画面・権限を列挙すること。
4. 実装 → テスト → 動作確認（UIは必ずブラウザ確認）の順で進めること。

### 3.2 勝手にやらないこと

- 新しいパッケージの追加
- Supabaseのスキーマをmigration経由以外で変更
- 要件定義にない機能の追加（"これもあったら便利そう" で増やさない）
- デザイントークン外の色・角丸・影の追加
- 本番DBへの破壊的操作

---

> **最後に**：グラストの業務システムは「現場の運用を止めずに回す」ための道具です。綺麗なUIで自己満足するより、**現場が使いたくなる・運用が止まらない**ことを最優先にしてください。
