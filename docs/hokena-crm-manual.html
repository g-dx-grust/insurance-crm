01. システム概要

何のためのシステムか

HOKENA CRM は、生命保険・損害保険代理店の業務オペレーションを止めずに、コンプライアンス要件を満たしながら回すための業務 CRM です。 グラスト G-DX 共通ルール（業務 SaaS としての落ち着いた UI / 監査ログ重視 / 運用負荷優先）に従って実装されています。

主なゴール：

  1. 顧客と契約の一元管理 — 顧客マスタ・家族構成・契約・特約・更改履歴を一画面でたどれる
  2. 保険業法対応の証跡化 — 意向把握記録は承認フロー付きで物理削除禁止
  3. 満期・タスク漏れの防止 — 満期 60 日前アラート、ダッシュボード／レポート／通知連携
  4. 代理店経営の見える化 — 月次推移・担当者別実績・MDRT 進捗・商品カテゴリ別構成
  5. Lark 連携前提の運用 — SSO、承認フロー、通知 Bot、Base 同期を将来一体運用

実装上の重要な特徴：

  • 論理削除（deleted_at） が基本。顧客・契約・案件は復元可能、意向把握は復元すら原則しません。
  • tenants.settings に JSONB で代理店ごとの設定を保持。Lark 連携設定もすべて DB 管理（ハードコード禁止）。
  • updated_at 自動更新トリガー が主要テーブルに設定済み。

このシステムが「やらないこと」

導入検討時の判断材料として、明示的に対象外としているスコープを示します。

  • 保険会社の見積もり機能・申込書作成は 行いません（外部の保険会社システム想定）
  • 顧客との直接的なメッセージング機能は 行いません（Lark / メール / 電話を運用で使う想定）
  • 会計・税務処理は 行いません（精算データのエクスポートまで）

02. はじめに — ログインと共通操作

ログイン

  1. ブラウザで https://<代理店ドメイン>/login にアクセスします。
  2. ［ID／パスワード］ にメールアドレスとパスワードを入力し、［ログイン］ をクリックします。
  3. ログインに成功するとダッシュボードに遷移します。

⚠️ ［Lark アカウントでログイン］ ボタンは現状無効化されています。Lark Developer Console での登録と環境変数設定が完了すると有効化されます（12. Lark 連携 参照）。

初回ログイン

  • 管理者が招待メールを送ると、招待リンク経由でパスワード設定後に通常ログインが可能になります。
  • パスワードを忘れた場合は管理者に再発行を依頼してください。

ログインできない場合

症状原因の例対応
「ログインに失敗しました」パスワード誤り／メールアドレス誤り入力内容を再確認
「このユーザーにはプロファイルが登録されていません」Supabase Auth は通ったが user_profiles が未作成管理者に連絡
ログイン後すぐにログイン画面に戻るセッション Cookie がブロックされている／別ドメインで保存されているブラウザの Cookie 設定を確認、シークレットウィンドウで再試行

  • ホーム / 顧客・契約 / 案件・活動 / 保険業法対応 / 精算・成績 / 分析・設定
  • 左下のアイコンで 開閉トグル ができます。折りたたみ状態はブラウザに記憶されます (hokena_sidebar_collapsed)。

顧客を編集する／削除する

編集

詳細画面右上の ［編集］ から入ります。入力フォームは新規登録と同じ項目です。

削除

詳細画面右上の ［削除］ から論理削除します。

  • 削除すると一覧から非表示になります（deleted_at がセットされる）。
  • データ自体は残るため、技術担当の操作で復元が可能です。
  • 削除した顧客に紐付く契約・案件・対応履歴・家族・意向把握は そのまま残ります（参照関係は維持）。

⚠️ 削除は確認モーダルで承認が必要です。誤操作防止のため、削除直前に顧客名を再表示します。

ユーザー管理

新しいスタッフを迎え入れる

  1. ［設定］ → ［ユーザー管理］ タブ → ［ユーザーを招待］
  2. メールアドレス、氏名、ロールを入力 → 招待送信
  3. スタッフがメール内リンクをクリック → パスワード設定
  4. ログイン → 担当者として顧客・契約に紐付け可能

スタッフが退職する

  1. ［設定］ → ［ユーザー管理］ → 該当ユーザー行 → ［無効化］
  2. ログイン不可、担当選択肢から除外
  3. 過去の対応履歴・担当履歴・監査ログは残ります

⚠️ 「物理削除」はできません。退職者の アカウント削除 ではなく、必ず「無効化」で対応してください。データの整合性が保たれます。

アクセス制御の二重構造

UI ガードだけではなく、データベースの Row Level Security (RLS) で多重に防御しています。

┌─────────────────────────────────────────────────────┐
│  ① UI ガード   ロールに応じて画面・ボタンを出し分け    │
└──────────────────┬──────────────────────────────────┘
                   │ URL 直叩きでも
                   ▼
┌─────────────────────────────────────────────────────┐
│  ② Server Action  Server 側で再度ロール確認           │
└──────────────────┬──────────────────────────────────┘
                   │ 例外的に通過しても
                   ▼
┌─────────────────────────────────────────────────────┐
│  ③ Supabase RLS  DB レベルで tenant_id + ロールを    │
│                  最終チェック (`current_user_role`)   │
└─────────────────────────────────────────────────────┘

テナント分離

すべての業務テーブルに tenant_id カラムがあり、RLS で以下のポリシーが適用されます：

USING ( tenant_id = current_user_tenant_id() )

current_user_tenant_id() は ログインユーザーの user_profiles から取得 される SQL 関数。他テナントのデータは原理的に参照不可。
