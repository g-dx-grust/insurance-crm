# HOKENA CRM 現状報告・打ち合わせ確認事項

作成日: 2026年5月11日

この資料は、先方打ち合わせで「現在どこまでできているか」「追加要望の可否」「完成に必要な情報」を確認するためのサマリーです。
詳細な画面操作は `HOKENA CRM 説明書` を参照してください。

## 現状サマリー

| 項目 | 状態 | 補足 |
|---|---|---|
| デプロイ | Vercel へ反映済み | 本資料はリポジトリ確認ベースで整理 |
| フロントエンド | 実装済み | Next.js 16 / React 19 / Tailwind CSS v4 |
| データベース | 実装済み | Supabase PostgreSQL、テナント分離 RLS、論理削除 |
| 認証 | ID + パスワードは実装済み | Lark OAuth は準備中 |
| 説明書 | 作成済み | 画面付き手順書を HTML / PDF 化 |
| Lark 連携 | 設定 UI と DB 保存は実装済み | 実 API 連携は段階的に有効化予定 |

## 現在利用できる主な機能

| 機能 | 現状 |
|---|---|
| ダッシュボード | KPI、月次推移、担当者別実績、満期アラートを表示 |
| 顧客管理 | 顧客検索、登録、編集、詳細、家族構成、対応履歴、高齢者フラグ |
| 契約管理 | 契約登録、編集、特約、更改履歴、満期管理、論理削除 |
| 案件管理 | カンバン、リスト、案件登録、ステージ変更、活動履歴、失注理由 |
| 意向把握 | 4ステップウィザード、比較推奨、チェックリスト、承認ステータス管理 |
| 精算・MDRT | CSV インポート、契約突合、入金確認、月次手数料、MDRT 進捗 |
| カレンダー | 月表示、週表示、予定登録、編集、削除、顧客・案件との紐付け |
| レポート | 月次契約推移、担当者別、商品カテゴリ別、顧客年代分布 |
| 設定 | 代理店情報、ユーザー招待、ロール変更、通知設定、コンプライアンス設定 |
| Lark 連携設定 | SSO / Base / Calendar / Approval / Bot / 通知テンプレートの設定保存 |

## 準備中・追加判断が必要な機能

| 機能 | 現状 | 打ち合わせで決めたいこと |
|---|---|---|
| Lark OAuth ログイン | ルートはあるが実ログインは未接続 | Lark アプリ情報、許可スコープ、ログイン運用 |
| Lark Base 同期 | 設定 UI は実装済み、同期 API は未接続 | Base / Table / Field 構成、同期方向、競合時の扱い |
| Lark Calendar 同期 | CRM 側カレンダーは実装済み、Lark 同期は未接続 | Calendar ID、同期対象、担当者別運用 |
| Lark Approval | CRM 内承認は実装済み、Lark 承認フロー起動は未接続 | 承認フロー ID、承認者、差戻し運用 |
| Lark Bot 通知 | 通知ログ・テンプレートは準備済み、送信 API は未接続 | 通知先チャット、送信タイミング、文面 |
| メール通知 | 準備中 | メール通知を使うか、Lark に一本化するか |
| レポート出力 | グラフ閲覧のみ | CSV / PDF エクスポートの必要範囲 |
| 監査ログ閲覧 UI | DB 側の監査ログは想定、閲覧画面は未実装 | 管理者画面で見る必要があるか |
| パスワード再発行 UI | アプリ内ボタンは未実装 | Supabase 側運用か、管理画面に追加するか |
| MDRT 目標設定 UI | 目標値は DB 管理、画面更新は未実装 | 管理者が年度目標を画面で変える必要があるか |
| タスク管理 | 旧タスクテーブルは撤去済み | カレンダー予定で足りるか、独立タスクが必要か |
| Web会議 URL 専用項目 | 現状はメモ欄で運用 | 専用フィールドを追加するか |

## 機能追加の可否整理

| 区分 | 内容 |
|---|---|
| すぐ運用確認できる | 顧客、契約、案件、意向把握、精算、カレンダー、レポート、設定 |
| 情報が揃えば対応可能 | Lark 実連携、既存データ移行、保険会社別 CSV テンプレート確定、レポート出力 |
| 仕様決定後に対応 | 独立タスク管理、監査ログ閲覧 UI、MDRT 目標設定 UI、通知ルール拡張 |
| 原則スコープ外 | 保険会社の見積作成、申込書作成、顧客との直接チャット、会計・税務処理 |

## 完成に必要な情報

### スタッフ・権限

| 必要情報 | 用途 |
|---|---|
| 氏名、メールアドレス、部署 | ユーザー招待、担当者選択 |
| ロール | `admin` / `agent` / `staff` の権限付与 |
| 有効・無効ステータス | 退職者や閲覧停止ユーザーの管理 |
| Lark ユーザー情報 | Lark OAuth / DM 通知を行う場合に必要 |

### 保険会社・商品・サービス

| 必要情報 | 用途 |
|---|---|
| 取扱保険会社一覧 | 契約登録、精算 CSV 照合 |
| 商品カテゴリ | 現状候補: 生命保険 / 損害保険 / 医療保険 / 介護保険 / 年金保険 |
| 商品名・サービス名 | 契約・案件・レポートの選択肢 |
| 手数料率、計上ルール | 精算・MDRT 管理 |
| 保険会社別 CSV サンプル | インポートテンプレート確定 |

### 顧客・契約データ

| 必要情報 | 用途 |
|---|---|
| 既存顧客データ | 初期移行 |
| 既存契約データ | 契約・満期アラート・精算突合 |
| 証券番号の採番ルール | 重複防止、CSV 照合 |
| 顧客ステータス | 見込 / 既存 / 休眠 で足りるか確認 |
| 家族情報の有無 | 高齢者対応、受取人・被保険者管理 |

### コンプライアンス・運用ルール

| 必要情報 | 用途 |
|---|---|
| 高齢者判定年齢 | 初期値は 70 歳 |
| 意向把握チェックリスト | 現状 9 項目で足りるか |
| 承認者と承認条件 | 意向把握の承認フロー |
| データ保持年数 | 削除・監査ログ・バックアップ方針 |
| 担当者の閲覧範囲 | 自分担当のみ / テナント全体 |

### Lark 連携

| 必要情報 | 用途 |
|---|---|
| Lark App ID / App Secret | Vercel 環境変数に登録 |
| Redirect URI | Lark Developer Console に登録 |
| 必要スコープ | SSO、Base、Calendar、Approval、Bot 通知 |
| Base ID / Table ID / Field ID | 顧客・契約・案件同期 |
| Calendar ID | 予定同期 |
| Approval Flow ID | 意向把握承認 |
| 通知先チャット ID | Bot 通知 |
| Bot 追加状況 | 通知先グループにアプリを追加する必要あり |

### 本番運用

| 必要情報 | 用途 |
|---|---|
| 本番ドメイン | リダイレクト URI、利用者案内 |
| Supabase 本番プロジェクト情報 | DB、Auth、バックアップ |
| Vercel 環境変数 | Supabase / Lark / APP URL |
| 初期管理者 | 初回ログイン、ユーザー招待 |
| 受入テストシナリオ | 完成判定 |

## 打ち合わせでの確認順

1. 現行画面を見せ、利用できる機能範囲を確認する
2. Lark 連携をどこまで初回リリースに含めるか決める
3. スタッフ・商品・保険会社・既存データの提供形式を確認する
4. 意向把握、承認、高齢者対応、通知の運用ルールを確認する
5. 追加要望を「初回リリース必須」と「リリース後改善」に分ける
6. 受入テストの日程、確認担当、完了条件を決める
