## 1. 基本方針と前提

G-DXシステムにおけるLark連携は、単なるデータソースとしての利用にとどまらず、**ユーザーがLarkとG-DXシステムをシームレスに行き来できる体験（シームレスなUX）**を提供することを目的とします。

- **APIで深く連携する必要がある箇所については、将来的なLark連携の可能性を考慮した設計とすること。**
- 環境変数は最小限（App ID/Secret等）に留め、運用中に変更される可能性のあるID類はデータベースで管理すること。

---

## 2. Lark OAuth認証とユーザープロフィール

LarkアカウントによるOAuth 2.0認証を実装する際、単にログインさせるだけでなく、システム内のUXを向上させるためにユーザー情報を最大限活用します。

### 2.1 プロフィール画像の取得と表示

Larkユーザーでログインした際、Lark上のアカウントプロフィール画像（アバター）や氏名をシステム内にも同期・表示させます。これにより、ユーザーは「自分がログインしている」という安心感を得られます。

**【実装ルール】**
- OAuthの `authen/v1/user_info` APIから `avatar_url`, `avatar_thumb`, `avatar_middle`, `avatar_big` などを取得します。
- 取得した画像URLは、システム内のデータベース（例: Supabaseの `users` テーブルの `avatar_url` カラム）に保存・キャッシュします。
- ヘッダーのユーザーアイコンや、コメント・ログの投稿者アイコンとしてシステム全体で一貫して表示します。

```typescript
// 取得例（g-dx_kanri の実装に準拠）
const avatarUrl = larkUser.avatar_big || larkUser.avatar_middle || larkUser.avatar_thumb || larkUser.avatar_url || null;
const displayName = larkUser.name || larkUser.en_name || 'Lark User';
```

### 2.2 Open IDの管理
- Larkの `open_id` はシステム内のユーザーを一意に識別するキーとして使用し、メールアドレスの変更などにも耐えられるように設計します。

## 3. 連携設定のシステム内管理（ハードコード禁止）

Larkの各種ID（Base ID、Table ID、チャットIDなど）は、運用フェーズで頻繁に変更される可能性があります。これらをソースコードや環境変数にハードコードすることは**厳禁**です。

### 3.1 データベースによる設定管理

Lark連携に関連する設定値は、システム内のデータベース（例: `system_settings` や `integration_configs` テーブル）で管理し、管理画面からUI経由で変更できる設計とします。

**【管理画面で設定できるようにすべき項目の例】**
- **通知先チャットID**: エラー通知や申請通知を送るLarkグループチャットのID（`chat_id`）
- **連携先 Base ID / Table ID**: データの読み書き先となるLark BaseのID
- **部署同期設定**: 同期対象とするルート部署のID

### 3.2 UI設計の要点
- 管理画面の「Lark連携設定」メニューに集約します。
- 各設定項目には、どこからそのIDを取得すればよいかのヘルプテキスト（またはツールチップ）を必ず添えます。

## 4. カスタムボットと通知（Lark IM連携）

システムからの通知（承認依頼、エラーアラートなど）をLarkに送信する際、カスタムボットを利用します。

### 4.1 カスタムボット追加の注意点明記

システムから特定のグループチャットに通知を送る場合、**そのチャットグループに事前にカスタムボット（アプリ）を追加しておく必要があります。** ボットが追加されていないとAPIがエラー（権限不足やチャット未検出）を返します。

**【実装ルール】**
- チャットIDを設定する管理画面のUI内に、**「通知を送信するためには、対象のLarkグループチャットに本システムのボット（アプリ）を追加してください」**という注意書きを必ず記載します。
- スペースの都合がある場合は、インフォメーションアイコン（ℹ️）のホバー表示（ツールチップ）でも構いません。

### 4.2 通知処理のベストプラクティス
- **非同期処理**: Lark APIへの通知送信は非同期（キューやバックグラウンドジョブ）で行い、ユーザーの画面操作をブロックしないようにします。
- **リトライ機構**: ネットワークエラー等に備え、失敗した通知はデータベースの `notification_logs` に記録し、指数バックオフによるリトライ（例: 5分後、15分後、60分後）を行う設計とします（G_HRの実装に準拠）。
- **ダイレクトメッセージ**: 個人宛の通知は `open_id` を用いてダイレクトメッセージとして送信します。

## 5. Lark Baseデータ連携

Lark Baseをデータソースとして利用する場合のルールです。

### 5.1 サーバーサイドプロキシの利用
- クライアント（ブラウザ）から直接Lark APIを呼び出すことはセキュリティ上禁止です。
- 必ずNext.jsのAPI RoutesやServer Actionsを経由し、サーバーサイドでTenant Access Token（またはApp Access Token）を付与してLark APIと通信します。

### 5.2 トークンのキャッシュ
- `tenant_access_token` や `app_access_token` は有効期限（通常2時間）があるため、都度取得するのではなく、Redisやインメモリキャッシュ、データベースを用いて有効期限内はキャッシュを再利用する設計とします。

---

## 6. セキュリティと権限管理

- **App ID / App Secret の管理**: これらシステム全体の根幹に関わる機密情報はデータベースには保存せず、必ずサーバーの環境変数（Vercel等のSecret設定）で管理します。
- **権限の最小化**: Lark開発者コンソールでアプリに付与する権限（Scope）は、システムが必要とする最小限のもの（例: `contact:user.base:readonly`, `im:message:send_as_bot` など）に留めます。
